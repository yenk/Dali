# Contributing

## Tier 1 — Canonical Case Corpus contributions

The Canonical Case Corpus is the primary benchmark track. A record belongs here if a court has publicly documented the AI-assisted citation failure — sanctions orders, judicial opinions, or filings with docket references are the accepted source types.

To propose a new corpus record, open a PR with:

1. A new entry in `data/internal/citation_failure_cases.json` following the `CitationFailureCase` schema in `corpus/schema.py`
2. A verifiable `source_url` (PACER, court opinion site, or news writeup citing the underlying docket) — records without one cannot count toward scoring
3. All ten required fields populated: `source_url`, `retrieval_date`, `source_type`, `incident_name`, `alleged_generated_citation`, `actual_status`, `failure_class`, `ground_truth_notes`, and correct taxonomy values from `corpus/taxonomy.py`
4. A brief PR description covering: what the court record establishes, why the failure class assignments are correct, and your retrieval date

Run the validator before opening the PR:

```bash
python -m corpus.validator data/internal/citation_failure_cases.json
```

Expected output: `N records loaded, M scoring-eligible` with no provenance or lineage errors.

Attorney names should be placed in `attorney_names_internal` only — never in free-text fields. The public artifact is generated by `corpus/anonymizer.py` and strips that field automatically.

## Commit attribution

Commits merged into the default branch must use your own Git author identity. Do not leave AI assistants as the Git author and do not add bot `Co-authored-by` trailers for Cursor or Claude, because GitHub can count those identities as repository contributors.

Before pushing, verify the latest commit metadata:

```bash
git log -1 --format='Author: %an <%ae>%n%B'
```

## Tier 2 — Synthetic probe contributions

Synthetic probes are controlled prompts that stress-test models and retrieval systems against known failure patterns. They are separate from Tier 1 canonical cases and live under `synthetic/`.

To add synthetic prompts, open a PR with:

1. New entries in the appropriate JSONL file under `synthetic/`
2. A brief justification in the PR description covering:
   - Why the prompt tests a meaningful failure mode
   - Why it is neutral (no phrasing that favors a model's training distribution)
   - Which difficulty tier: `known_case`, `emerging_area`, or `adversarial`
   - If derived from a Tier 1 case, the `parent_incident_id` and `mutation_type`

### Prompt format

```json
{
  "id": "legal_case_009",
  "category": "legal",
  "subcategory": "case_citations",
  "prompt": "...",
  "difficulty": "known_case",
  "notes": "Optional. What this prompt is testing."
}
```

IDs must be unique across all files. Use `{subcategory}_{NNN}` format with zero-padded three-digit suffix.

### Neutrality standard

- No leading questions ("Is it true that the Supreme Court held...")
- No prompts that embed the answer ("Cite the case where X was established")
- No prompts targeting a single model's known weakness
- Cover a real legal domain, not a synthetic edge case

## Result contributions

External run results are welcome.

**Tier 1 results** (local reference evaluator — no API key required):

```bash
python runners/run_integrity.py \
    --corpus data/public/citation_failure_cases.json \
    --output results/v0.2/{your-run-date}/integrity.json
```

Open a PR adding the output JSON to `results/v0.2/{your-run-date}/`. Include the `policy_version` field from the output — results from different policy versions are not aggregated without `--allow-cross-version`.

**Tier 2 results** (requires model API access):

```bash
python runners/run_synthetic.py \
    --models <model-id> \
    --prompts synthetic/ \
    --output results/v0.2/{your-run-date}/synthetic.json
```

Include the `methodology.json` produced by the runner, the model version string, API endpoint, and run date.

Result files are immutable once merged — no editing after merge.

## Bug reports

File an issue. Label it appropriately:

- **Corpus annotation bug** — wrong `failure_class`, incorrect `actual_status`, bad `source_url`, or factual error in `ground_truth_notes`. Include the `case_id` and what the record should say.
- **Runner bug** — unexpected output from `run_integrity.py` or `run_synthetic.py`. Include the `case_id` or prompt ID and the full result JSON.
- **Schema or validator bug** — validation rejects a valid record, or accepts an invalid one. Include the record JSON and the validator output.
